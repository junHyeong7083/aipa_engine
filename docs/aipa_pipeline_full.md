# AIPA Engine — 기술 파이프라인 전체 설명서

> 본 문서는 실제 코드(`src/aipa_engine`, `rag/`, `training/`)를 근거로 작성되었습니다.
> 각 섹션은 슬라이드 1~2장 단위로 구성되어 있어 그대로 PPT로 옮길 수 있습니다.

---

## 1. 한 줄 요약 + 서비스 구성

**한 줄 요약**
AIPA Engine은 한국 공공통계(KOSIS)로 만든 "가상 응답자 패널"에 자체 임베딩 모델·소형 LLM·검색 트렌드(RAG)를 결합해, 실제 사람처럼 응답하는 설문/반응 시뮬레이션을 제공하는 엔진입니다.

**서비스 구성 (2종)**

| 서비스 | 설명 | 핵심 엔진 |
|---|---|---|
| **설문 시뮬레이션** | 가상 페르소나 패널이 설문 문항에 응답 → 통계 보정된 결과 리포트 | PopulationGenerator + ResponseGenerator + Calibrator |
| **AI 채팅 / 자극물 평가** | 광고·제품 등 자극물에 대한 페르소나별 다축 평가, 대화형 반응 | EvalService(임베딩 점수) + 0.5B 이유생성 + (채팅 한정) 외부 LLM |

**핵심 설계 원칙**
- **판단(점수)은 자체 임베딩 모델**, **표현(이유 문장)은 자체 0.5B 모델**이 담당
- 외부 API(Claude)는 **채팅·폴백 경로**에서만 보조적으로 사용
- 모델이 없거나 실패해도 동작하도록 **단계별 폴백 체인** 내장

## 3. 설문 1건이 처리되는 단계별 흐름

`ResponseGenerator.generate_response()` 기준, 페르소나 1명이 문항 1개에 답하는 과정입니다.

```
[입력: 페르소나 + 설문 문항]
        │
        ▼
① 무응답 체크 (_check_nonresponse)
   연령별 기본 무응답률 + 민감 키워드(소득·정치 등)면 2배
   → 걸리면 "(무응답)" 반환하고 종료
        │ (통과)
        ▼
② 직선응답 체크 (_check_straightlining)
   문항 5개 이상 진행 + 피로도 누적 시 이전 답 반복
   → 걸리면 직전 답변 그대로 반환
        │ (통과)
        ▼
③ 임베딩 점수 예측 (_predict_score_with_embedding)
   페르소나(연령·성별·직업·특성) + 질문 카테고리·축 → 0~100점
   (CPU, 약 0.01초)
        │
        ▼
④ RAG 트렌드 보정 (검색량 증감 → 점수 가감)  ※ 7번 섹션 참고
        │
        ▼
⑤ 점수 → 선택지 확률 변환 (_score_to_choice_index)
   가우시안 커널: 점수가 높을수록 긍정 선택지에 확률 집중
        │
        ▼
⑥ 8대 인간 응답 편향 적용 (순서 고정)
   극단회피 → 묵종 → 사회적바람직성 → 그룹동조 → 시간대효과
        │
        ▼
⑦ 확률 정규화 후 샘플링 (np.random.choice)
   → 최종 선택지 결정 + 그룹/직선응답 기록 갱신
        │
        ▼
⑧ 이유 생성 (_generate_explanation_local)
   자체 0.5B 모델로 1문장 근거 생성
        │
        ▼
[출력: 선택지 + 확률 + 이유 문장]
```

**폴백 체인 (장애에 강한 설계)**

| 단계 | 정상 경로 | 실패 시 폴백 |
|---|---|---|
| 점수 예측 | 자체 임베딩 모델 | → 확률 엔진(`_calculate_choice_probabilities`, 연령별 가격/기술 민감도 + 특성-키워드 매핑) |
| 이유 생성 | 자체 0.5B 모델 | → 템플릿 문장(점수대별 high/mid/low 풀에서 선택) |
| RAG 보정 | ChromaDB 트렌드 | → 보정값 0 (점수 그대로 사용) |

> 즉, 모델/DB가 없는 환경에서도 엔진은 멈추지 않고 합리적인 결과를 냅니다.

## 4. 공공 데이터셋 → 페르소나 생성

**사용 공공 통계 (KOSIS OpenAPI)**

| 용도 | 통계표 ID | 통계명 | 추출 항목 |
|---|---|---|---|
| 연령·성별 분포 | `DT_1BPA001` | 장래인구추계 (성·연령별 추계인구) | 연령대별/성별 인구 비율 |
| 직업 분포 | `DT_1DA7E27S_NEW` | 경제활동인구조사 (직업별 취업자) | 직업 대분류별 취업자 비율 |

- 기관 ID `101`(통계청), 5세 단위 연령 코드를 10대~60대+ 그룹으로 집계
- **캐싱 전략**: 결과를 `data/processed/*.json`에 저장, **30일** 이내면 재사용 → API 호출 최소화
- **폴백**: API 키 없음/호출 실패 시 하드코딩된 2023~2024 기준 기본 분포 사용

**페르소나 생성 알고리즘 (`PopulationGenerator`)**

1. **한계분포 샘플링 (Marginal)**
   - 연령대: 10대 9% / 20대 13% / 30대 14% / 40대 16% / 50대 17% / 60대+ 31%
   - 성별: 남 49.9% / 여 50.1%
   - `numpy.random.choice`로 분포에 맞게 확률적 추출
2. **조건부확률 샘플링 P(직업|연령)**
   - 예: 10대 → 학생 85%, 20대 → 학생 35%/회사원 30%, 60대+ → 무직 35%/자영업 20%
   - 데이터 근거: `data/processed/conditional_probs.json` (MDIS 경제활동인구조사 기반 P(직업|연령), P(직업|성별))
3. **연령대별 특성 풀 (`AGE_TRAIT_POOL`)**
   - 연령에 맞는 특성만 부여(예: 10대→트렌디/SNS활발, 60대+→건강최우선/보수적)
   - 각 페르소나에 2~4개 특성 랜덤 부여
4. **한국식 이름 생성** (성씨 + 성별별 이름 조합)

**→ 핵심: 비현실 조합 방지**
단순 랜덤이 아니라 "연령에 맞는 직업·특성"만 나오도록 조건부확률을 적용 → "60대 틱톡커", "10대 임원" 같은 비현실 페르소나를 구조적으로 차단합니다.

> SNS 플랫폼 시뮬레이션 시에는 플랫폼별 프로필(8번 박스 참고)로 연령/성비/직업/특성 분포를 교체합니다.

## 5. 8가지 인간 응답 편향

설문 방법론 연구를 코드로 구현한 8가지 편향. 단순 랜덤 응답과의 결정적 차이점입니다.
(연령대별 강도 테이블 `AGE_RESPONSE_TRAITS`로 차등 적용)

| # | 편향 | 1줄 설명 |
|---|---|---|
| 1 | **극단 회피(중심화 경향)** | 1번/5번 같은 극단 선택을 피하고 중앙으로 확률을 옮김 (한국인 특히 강함) |
| 2 | **묵종 편향** | "그렇다" 쪽(긍정 보기)에 동의하는 경향, 고연령일수록 강함 |
| 3 | **사회적 바람직성** | 환경·건강·기부 등 바람직한 주제에서 실제보다 더 긍정적으로 답함 |
| 4 | **직선응답(피로도)** | 문항 5개 이상부터 피로 누적 시 직전 답을 그대로 반복 |
| 5 | **무응답** | 일정 확률로 문항 스킵, 소득·정치 등 민감 키워드면 스킵률 2배 |
| 6 | **그룹 동조(또래 효과)** | 같은 연령·성별 그룹의 최근 응답 평균 쪽으로 확률 끌림 |
| 7 | **시간대 효과** | 야간(22~6시) 극단화, 월요일 오전 부정, 금요일 오후 긍정 경향 |
| 8 | **점수 노이즈** | 같은 사람도 매번 다른 점수(연령별 ±4~8점 정규분포 변동) |

> 적용 순서: 극단회피 → 묵종 → 사회적바람직성 → 그룹동조 → 시간대효과 (코드상 고정 순서)

## 6. 자체 모델 2종

> 외부 LLM(Claude)에 매 응답을 묻지 않고, **자체 모델로 판단·표현을 모두 처리**하는 것이 비용·속도·일관성의 핵심입니다.

### (A) 임베딩 점수예측 모델 — "판단"

- **역할**: 페르소나 × 자극물(카테고리·평가축) → 0~100점 선호/긍정도 예측
- **구조**: 연령·성별·직업·특성·카테고리·축을 각각 임베딩 → 특성은 어텐션 가중합 → MLP 인코더 → Sigmoid 점수 출력 (`PersonaEmbeddingModel`)
- **성능**: **CPU에서 약 0.01초**, GPU·외부 API 불필요 → 수천 건도 저비용 처리
- **위치**: `training/models/embedding/persona_embedding_model.pt`

### (B) Qwen2.5-0.5B LoRA 이유생성 모델 — "표현"

- **역할**: 선택한 답변/점수에 대한 **페르소나 관점의 1문장 근거** 생성
- **베이스**: `Qwen/Qwen2.5-0.5B-Instruct` + **LoRA** 어댑터 파인튜닝 → 병합본 사용
- **학습 방식**: **Knowledge Distillation** — 대형 모델(Claude)이 만든 **약 3천 건 규모의 합성 데이터**로 소형 모델을 학습
- **추론**: CPU(float32), 위치 `training/models/reasoning/merged`

### 역할 분담 요약

| 기능 | 담당 | 외부 API |
|---|---|---|
| 점수/선호도 판단 | 자체 임베딩 모델 | ✕ |
| 이유 문장 표현 | 자체 0.5B 모델 | ✕ |
| 대화형 채팅 | 외부 LLM | ○ (채팅에 한정) |
| 모델 부재 시 폴백 | 확률 엔진 + 템플릿 (+필요시 외부 LLM) | △ |

## 7. RAG 트렌드 보정 (검색량 증감을 관련도 가중으로 반영)

> 한 줄 정의: **검색량 증감을 관련도 가중으로 반영하고, 평가축·페르소나별로 차등 적용하는 트렌드 보정.**

### 데이터 인덱스 (`rag/build_index.py`)
- **임베딩**: `jhgan/ko-sroberta-multitask` (한국어 문장 임베딩)
- **저장소**: ChromaDB, 컬렉션 2종
  - `naver_trends` — 네이버 검색/쇼핑 트렌드
  - `kosis_stats` — 공공통계
- **핵심 메타데이터**: 각 트렌드 문서에 `latest_ratio`(최근 검색량 비율), `avg_ratio`(기간 평균)를 저장
- 증분 인덱싱(`--incremental`) 지원

### 보정 로직 (`EvalService` — 업그레이드 버전)

**1) 관련도 가중 (문서 개수 아님)**
- 검색 결과를 벡터 거리(distance)로 가중: `relevance = max(0, 1 − distance/threshold)`
- 가까운 문서일수록 높은 가중치 → "몇 건 나왔나"가 아니라 "얼마나 관련 있나"로 신호 산출

**2) 실제 검색량 증감을 방향으로 사용**
```
pct = (latest_ratio − avg_ratio) / avg_ratio      # 검색량 증감률
pct = clamp(pct, −0.5, +0.5)                       # ±50%로 제한
weighted_trend = Σ(relevance × pct) / Σ(relevance) # 관련도 가중 평균
base_points    = weighted_trend × 30.0             # ≈ ±15점
```
- 검색량 **상승 → 점수 +**, **하락 → 점수 −** (메타데이터의 실제 수치 기반, 키워드 카운팅 아님)

**3) 평가축별 민감도 (`AXIS_TREND_WEIGHT`)**
- 강하게 반응: **트렌드부합(×1.2) · 구매의향/관심도(×1.0)**
- 약하게 반응: **신뢰도(×0.3) · 가격적절성(×0.2)** — 시장 분위기와 무관한 축은 덜 흔들림
- 매핑에 없는 축은 기본 ×0.4

**4) 페르소나별 민감도 (`_persona_trend_sensitivity`, 0.4~1.6배)**
- 연령: 10·20대 ×1.2 → 60대+ ×0.7
- 특성: "트렌디/얼리어답터/SNS" 등이면 +0.15, "보수/전통/절약/실용"이면 −0.15
- 최종 클램프: **×0.4 ~ ×1.6**

**최종 보정값**
```
축별 보정 = base_points × 페르소나_민감도 × 축_가중치
예) 구매의향 +6.2, 가격적절성 +1.1 (음수도 가능)
```

> ⚠️ 정직성: 이는 **검색 트렌드를 반영하는 설계**이며, "실시간"이나 "측정된 정확도 향상"을 주장하지 않습니다. 데이터 수집 주기에 따라 갱신됩니다.

## 8. 캘리브레이션 (IPF / Raking 사후보정)

- **문제**: 확률 샘플링이라 패널의 실제 분포가 목표 인구통계와 약간 어긋날 수 있음(예: 남자 과대표집)
- **해법**: **IPF(Iterative Proportional Fitting) / Raking**으로 응답 가중치를 조정 (실제 여론조사 표준 기법)

**동작 (`Calibrator`)**
1. 목표 한계분포 설정 (config 지정값 또는 실제 통계)
2. 변수(연령→성별→…)별로 `목표비율/현재비율` 배수를 가중치에 곱함
3. 가중치 변화가 `tolerance(1e-6)` 미만이 될 때까지 최대 100회 반복 → 수렴
4. 평균 가중치 1.0이 되도록 정규화 후, 각 응답에 가중치 부여

- **충실도 측정**: `calculate_distribution_fidelity()` — 목표 대비 |실제−목표| 평균 오차를 1에서 빼 0~1 점수화

> 효과: "샘플 편향"을 통계적으로 교정해, 가중 집계 결과가 목표 모집단 분포를 따르도록 만듭니다.

## 9. 기술 차별화 (ChatGPT 직접 질문 vs AIPA)

| 항목 | ChatGPT에 직접 질문 | **AIPA Engine** |
|---|---|---|
| 응답 주체 | 단일 모델의 "평균적" 답변 | KOSIS 분포 기반 **다수 가상 페르소나 패널** |
| 인구 대표성 | 없음 (모델 편향 그대로) | 연령·성별·직업 **공공통계 분포 반영** |
| 인간 응답 특성 | 거의 없음 | **8대 응답 편향** 시뮬레이션 |
| 비현실 조합 | 통제 불가 | 조건부확률 P(직업\|연령)로 **구조적 차단** |
| 트렌드 반영 | 학습 시점에 고정 | 검색량 증감 **RAG 보정(설계)** |
| 통계 보정 | 없음 | **IPF/Raking 사후보정** |
| 비용·속도 | 호출마다 API 비용 | 점수=임베딩(CPU 0.01초), 이유=0.5B **자체 모델** |
| 결과 형태 | 자유 텍스트 | 선택지 분포 + 가중치 + 다축 점수 + 근거 |
